# Executive Summary

The `preview` branch CI/CD pipeline is now GCP-first. Pull requests run web and
API checks only. Pushes to `preview` run those checks, publish Plane component
images to Google Artifact Registry, run backend migrations on GKE, roll out the
new images to the `plane-ce` namespace, and smoke `https://app.manut.xyz`.

# Business Goals

- Ship self-host Plane changes to the production GCP runtime automatically.
- Remove the deleted Railway environment from the release path.
- Keep production releases reproducible through immutable commit image tags.
- Fail clearly if GCP deploy credentials are not configured.

# Technical Goals

- Build component images in Artifact Registry:
  `asia-southeast1-docker.pkg.dev/affine-495114/affine`.
- Authenticate GitHub Actions to GCP with Workload Identity Federation.
- Deploy the same tag to API, worker, beat-worker, web, admin, live, and space.
- Run `./bin/docker-entrypoint-migrator.sh` before workload rollout.
- Verify the public instance endpoint after rollout.

# Requirements

- PRs targeting `preview` must run deterministic web and API checks.
- Pushes to `preview` must publish GCP Artifact Registry images after checks.
- Pushes to `preview` must deploy to GKE after image publication.
- GitHub Actions must not invoke Railway hooks or build Railway AIO images.
- Deployment must use repository variables:
  - `GCP_WORKLOAD_IDENTITY_PROVIDER`
  - `GCP_SERVICE_ACCOUNT`
- Optional repository variables may override defaults:
  - `GCP_APP_URL`
  - `GCP_ARTIFACT_REPOSITORY`
  - `GCP_PROJECT_ID`
  - `GCP_REGION`
  - `GKE_CLUSTER`
  - `GKE_NAMESPACE`

# Non-Goals

- Do not use GHCR as the production image registry.
- Do not publish to upstream DockerHub repositories.
- Do not store JSON service account keys in GitHub secrets.
- Do not replace Helm ownership of the Plane release in this change.

# Architecture

```txt
PR to preview
  -> web checks
  -> API checks

push to preview
  -> web checks
  -> API checks
  -> build component images to Artifact Registry
  -> run GKE migration job
  -> kubectl set image for Plane workloads
  -> wait for rollout status
  -> smoke https://app.manut.xyz/api/instances/
```

# Data Models

No application data model changes are introduced by the pipeline itself.
Application migrations are executed through the backend image before rollout.

Image tags:

- `asia-southeast1-docker.pkg.dev/affine-495114/affine/plane-frontend:preview`
- `asia-southeast1-docker.pkg.dev/affine-495114/affine/plane-frontend:preview-<short_sha>`
- Equivalent branch and commit tags for backend, admin, live, proxy, and space.

# API Contracts

No runtime API contracts change. The deployment smoke check requires
`GET /api/instances/` on `GCP_APP_URL` to return a successful response.

# Security

- GitHub Actions authenticates through GCP Workload Identity Federation.
- The deployer service account should have only Artifact Registry writer,
  cluster access, and namespace-scoped Kubernetes rollout permissions.
- Workflow logs must never print secret values.
- Rollback uses prior immutable Artifact Registry tags or Kubernetes rollout
  history.

# Edge Cases

- Missing GCP auth variables fail the image/deploy jobs with an explicit error.
- Invalid Docker tag characters in branch names are normalized.
- A failed migration job prints job details and container logs before failing.
- A failed rollout stops before the smoke check.
- A newer push cancels stale runs for the same branch.

# Testing Strategy

- Parse the workflow YAML locally.
- Assert the active CI/CD workflow has no Railway references.
- Run `git diff --check`.
- Let GitHub Actions validate real Artifact Registry/GKE execution after push.

# Rollback Plan

- Revert the CI/CD commit if the workflow itself is faulty.
- Roll back Kubernetes deployments to the previous ReplicaSet:
  `kubectl rollout undo deployment/<name> -n plane-ce`.
- For image-level rollback, set workloads back to the previous
  `preview-<short_sha>` Artifact Registry tag.
- For database issues, prefer Cloud SQL PITR clone before restoring over
  production.

# Milestones

Goal: Railway no longer runs from `Plane CI/CD`.

Success Metrics:

- No `deploy_railway`, Railway hook, or Railway AIO job remains in
  `.github/workflows/ci-cd.yml`.

Rollback:

## Milestone 2 - Publish To Artifact Registry

Goal: Pushes to `preview` publish fork-built component images to GCP.

Success Metrics:

- Artifact Registry receives branch and commit tags for each Plane component.

Rollback:

- Redeploy the previous Artifact Registry commit tag.

## Milestone 3 - Deploy To GKE

Goal: Successful image publication rolls production GKE workloads.

Success Metrics:

- Migration job completes.
- All Plane deployments finish rollout.
- `GET /api/instances/` succeeds on `https://app.manut.xyz`.

Rollback:

- Undo deployments or set the previous known-good image tag.

# Acceptance Criteria

- CI/CD no longer references Railway.
- `preview` push builds GCP Artifact Registry images.
- `preview` push deploys those images to GKE when GCP auth variables are set.
- Deployment runs migrations before workload image updates.
- Deployment smoke check proves the live Plane instance responds.
