# Executive Summary

Add a fork-safe GitHub Actions CI/CD pipeline for the `preview` branch. Pull requests run deterministic web and API validation. Pushes to `preview` run the same gates, build Plane component images from this fork, assemble an all-in-one image from those component images, publish the images to GitHub Container Registry, and optionally trigger Railway through a deploy hook.

# Business Goals

- Make development safer by validating pull requests before merge.
- Make deployment repeatable by publishing immutable SHA-tagged images.
- Keep the current Railway preview path deployable without relying on upstream `makeplane/*` images.
- Avoid leaking or committing deployment secrets.

# Technical Goals

- Run frontend format, lint, type, and test gates with Node `22.18.0` and pnpm.
- Run backend unit tests through the existing Docker Compose test stack.
- Build and publish component Docker images to GHCR.
- Build and publish a Railway-ready AIO image that uses the fork-built component images.
- Support a stable branch tag plus an immutable commit tag for rollback.

# Requirements

- PRs targeting `preview` must run CI without requiring a reviewer assignment.
- Pushes to `preview` must run CI before image publishing.
- Image publishing must use `GITHUB_TOKEN` and `packages: write`.
- Deployment must be optional and secret-gated through `RAILWAY_DEPLOY_HOOK_URL`.
- Workflow concurrency must cancel stale runs per branch.
- The workflow must not publish to `makeplane/*` DockerHub repositories.

# Non-Goals

- Do not replace the existing upstream manual release workflow.
- Do not store Railway tokens, project IDs, database URLs, or app secrets in the repo.
- Do not mutate Kubernetes manifests or live GKE resources in this change.
- Do not make Railway wait on unrelated default security scans.

# Architecture

```txt
PR to preview
  -> web CI
  -> API CI

push to preview
  -> web CI
  -> API CI
  -> build component images to ghcr.io/<owner>/<repo>/*
  -> build AIO image from those component images
  -> optional Railway deploy hook
```

# Data Models

No application data model changes.

Image tags:

- `ghcr.io/<owner>/<repo>/plane-frontend:<branch>`
- `ghcr.io/<owner>/<repo>/plane-frontend:<branch>-<short_sha>`
- Equivalent tags for backend, admin, space, live, proxy, and AIO.

# API Contracts

No runtime API contract changes.

# Security

- Use `GITHUB_TOKEN` for GHCR publishing.
- Keep Railway deployment behind `RAILWAY_DEPLOY_HOOK_URL`.
- Do not echo secret values.
- Use immutable commit tags for rollback.
- Keep DockerHub publishing out of the automatic path.

# Edge Cases

- If `RAILWAY_DEPLOY_HOOK_URL` is not configured, the workflow publishes images and exits deployment successfully with an explicit message.
- If a branch name contains invalid Docker tag characters, the workflow normalizes it.
- If a newer commit arrives, older runs are cancelled by concurrency.
- If GHCR publishing passes but Railway deployment fails, the published SHA tag remains available for manual rollback.

# Testing Strategy

- Parse all workflow YAML files locally.
- Validate shell snippets compile with `bash -n` where practical.
- Run `git diff --check`.
- Let GitHub Actions validate real execution after push.

# Rollback Plan

- Revert the CI/CD commit.
- In Railway, redeploy the previous known-good SHA tag or the prior image.
- If the deploy hook causes issues, remove `RAILWAY_DEPLOY_HOOK_URL` from repository secrets to keep image publishing active while disabling automatic deploy.

# Milestones

## Milestone 1 - CI Foundation

Goal: PRs and pushes run web/API validation.

Success Metrics:

- Workflow starts on PRs to `preview`.
- Web and API jobs have clear cache and teardown behavior.

Risks:

- API Docker tests can be slow on GitHub-hosted runners.

Rollback:

- Disable or revert the workflow.

## Milestone 2 - Image Publishing

Goal: Pushes to `preview` publish fork-built component and AIO images.

Success Metrics:

- GHCR contains branch and SHA tags for every component and AIO image.

Risks:

- Docker build time can be high for the monorepo.

Rollback:

- Redeploy the previous SHA tag.

## Milestone 3 - Railway Deployment Trigger

Goal: Railway can redeploy after successful image publication.

Success Metrics:

- Deploy hook runs only after image publication.
- Missing deploy hook secret does not fail CI.

Risks:

- Railway service must be configured to pull the stable branch tag or redeploy from the desired source.

Rollback:

- Remove or rotate the Railway deploy hook secret.
